# 开发规范文档
## 1. 总体规范

- 所有配置集中放在 `E:\End-To-End_Recommendation_System_X\conf\application-local.yml`。
- 不在代码中硬编码 MySQL、Redis、Kafka、Milvus、模型路径、端口和密码。

- 主目录下只保留必要配置文件，避免各模块散落重复配置。

- 所有模块日志写入 `E:\End-To-End_Recommendation_System_X\logs`。
- 所有模型文件写入 `E:\End-To-End_Recommendation_System_X\data\model`。
- 所有原始数据写入 `E:\End-To-End_Recommendation_System_X\data\raw`。

## 3. 前端规范

### 3.1 命名

| 类型 | 规范 | 示例 |
| --- | --- | --- |
| 组件 | PascalCase | `ProductCard.vue` |
| 组合函数 | camelCase，以 use 开头 | `useExposureTracker.js` |
| Store | camelCase | `recommendationStore` |
| API 文件 | camelCase | `recommendation.js` |

### 3.2 状态管

- 推荐列表只放在 recommendmendation Store。
- 购物车状态只放在 cart Store。
- 用户 ID 和登录态只放在 user Store。
- 不在组件之间通过多层 props 传递全局状态。
### 3.3 埋点规范

- 任何埋点都必须包含 `userId`、`itemId`、`behaviorType`、`timestamp`、`requestId`、`scene`。
- 曝光埋点必须做单批次去重

- 埋点失败不得阻塞用户操作

- 行为类型只能使用 0 到 4。
### 3.4 UI 规范

- 风格参考得物：浅色背景、商品卡片清爽、强调图像和价格

- 主要按钮颜色保持克制，避免大面积高饱和色

- 商品卡片必须包含图片、标题、价格、收藏、点击、加购、购买。

- 移动端优先保证两列瀑布流可读性。
## 4. 后端规范

### 4.1 包命

- Controller 只做参数校验和响应转换。

- Application Service 做流程编排。

- Domain Service 放领域规则。

- Infrastructure 放数据库、Redis、Kafka、HTTP Client。
### 4.2 接口规范

- REST API 路径统一 `/api/v1` 开头。

- 响应统一使用 `code/message/requestId/data`。
- 每个推荐请求都必须生成或透传 requestId
- 参数错误返回 `400001`。
- 推理服务错误返回 `500201`。
### 4.3 日志规范

每条推荐刷新日志必须包含

- requestId
- userId
- scene、
- triggerType、
- candidateSize、
- modelVersion、
- latencyMs、
- fallbackType。
### 4.4 Redis 规范

- Key 必须带业务前缀

- 用户维度 Key 必须包含 userId
- 推荐候选必须设。TTL。
- 写候选队列时必须覆盖旧队列，避免新旧推荐混合。?
## 5. Python 算法规范

### 5.1 代码结构

- `algorithm/common` 放训练和推理共享逻辑

- `algorithm/recall` 只放召回模型

- `algorithm/ranking` 只放排序模型

- `algorithm/inference` 只放在线服务

- `algorithm/training` 放训练入口和任务编排。
### 5.2 注释要求

用户要求排序模型每行代码都加详细注释。实。MMoE 排序模型时应遵守

- 每个类说明输入、输出和用途。

- 每个张量变换说明维度

- 每个损失函数说明标签含义

- 每个防泄露逻辑说明时间边界。
### 5.3 特征一致性
- 不允许训练和推理各写一套特征处理。

- `feature_config.json` 是连续特征归一化和分桶的唯一配置来源

- `<UNK>` ID 必须固定

- 线上查不到 ID 时统一映射为 `<UNK>`。
### 5.4 数据泄露规范

禁止

-  Label_Time 之后的行为构造训练特征。

- 用购买结果直接作为输入特征。

- 排序负样本使用未来窗口判断未点击

- 离线评估时随机切分时间序列数据。

- 使用推荐请求之后的用户位置或标签偏好参与当前请求 LBS/标签召回。
必须

- 使用 Time Split：
- 在样本中保存 feature_time ≤ label_time。
- 训练前校。`feature_time < label_time`。
- LBS 特征、标签偏好特征也必须保存 feature_time。
### 5.5 LBS 与标签召回规范。
LBS 召回

- 用户 geohash 只能来自用户历史行为或用户当前请求显式携带的位置

- 用户位置缺失时跳过 LBS 通道

- 商品 geohash 缺失时不进入 LBS 倒排索引

- MVP 使用 geohash 前缀召回，后续可升级为经纬度距离过滤。
标签召回

- 商品标签统一写入 `rec_item_tag`。
- 用户实时标签偏好统一写入 Redis `feature:user:tag_pref:{userId}`
- 标签维度必须在配。`recommendation.tagRecallDimensions` 中声明。

- 新增标签维度时，需要同步更新离线特征、实。Flink 聚合和在线推理解析逻辑。

## 9. 用户与权限开发规

- 密码字段只能命名 `password_hash`，不得保存明文密码。

- 注册和重置密码都必须使用 BCrypt。
- Controller 不直接生。token，统一通过 `AuthService`。
- 管理员接口路径统一使用 `/api/v1/admin/**`。
- 管理员接口必须校。`ADMIN` 角色

- 业务接口需要用户身份时，优先从 Spring Security 上下文读取 `userId`
- 允许游客访问的接口必须显式标注，例如首页推荐和商品列表。

- 登录日志不得记录明文密码

- 返回用户信息时不得返回 `password_hash`。

## 10. 在线特征读取规范

- 在线推荐特征（sourcee of truth 以 Redis
- Python 推理服务是在线推荐特征的唯一读取方。- Spring Boot 不读取 `feature:user:*`、`rec:lbs:index:*`、`rec:tag:index:*` 作为推理输入

- 推理请求体不得携带 `userRealtimeFeatures` 这类实时特征整包

- 同一次推理请求应记录 `featureSnapshotTime`，用于排查特征快照和请求结果。
