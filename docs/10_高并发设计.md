# 高并发设计

## 1. 设计目标

| 维度 | 目标 |
| --- | --- |
| 推荐刷新 P99 | < 1.5 s（开发机），< 800 ms（4 核 16G） |
| Tomcat 接受连接 | 同时承载 ≥ 8000 长连接、并发请求 200+ |
| 推理服务故障 | 单接口 3 s 超时；连续失败率 ≥ 50% 时熔断 10 s |
| 登录暴力破解 | 单 IP 5 分钟 ≥ 30 次失败拒绝；单账号 5 次失败锁定 15 分钟 |
| 行为日志 | Kafka → 内存缓冲 → 批写 MySQL，崩溃时回滚到内存重试 |
| 资源隔离 | 推理、快照、Kafka flush 三个线程池独立，互不影响 |

## 2. 后端线程模型

```
                                ┌──────────────────────┐
HTTP 请求 ──> Tomcat (NIO)      │ Tomcat worker (400)  │
                                └────────┬─────────────┘
                                         │
                       ┌─────────────────┴──────────────────┐
                       │                                    │
              业务同步逻辑                            异步 @Async
        (DB / Redis 短任务 < 50ms)                  ┌──────────────────────┐
                       │                            │ snapshotExecutor 4-8 │  落 rec_request_snapshot
                       │                            └──────────────────────┘
                       │                            ┌──────────────────────┐
                       │                            │ kafkaFlushExecutor   │  批量写 rec_behavior_log
                       │                            └──────────────────────┘
                       ▼
              InferenceClient.recommendAsync
              ├── @CircuitBreaker(name=inference)
              ├── @TimeLimiter(name=inference, 3s)
              └── CompletableFuture.supplyAsync(executor)
                                                ┌──────────────────────────┐
                                                │ inferenceExecutor 32-64  │
                                                │ + queue=200, CallerRuns  │
                                                └──────────────────────────┘
                                                            │
                                                            ▼
                                       Apache HttpClient5 (pool 200 / 100 per-route)
                                                            │
                                                            ▼
                                          Python 推理服务 :9000
```

### 关键参数对应配置

| 项 | 配置键 | 默认 |
| --- | --- | --- |
| Tomcat 工作线程 | `server.tomcat.threads.max` | 400 |
| Tomcat backlog | `server.tomcat.accept-count` | 200 |
| Tomcat 连接数 | `server.tomcat.max-connections` | 8192 |
| 推理线程池 | `app.executor.inference.{core,max,queue}` | 32 / 64 / 200 |
| 快照线程池 | `app.executor.snapshot.{core,max,queue}` | 4 / 8 / 1000 |
| Kafka flush 线程池 | `app.executor.kafkaFlush.{core,max,queue}` | 2 / 4 / 50 |
| Hikari 连接池 | `spring.datasource.hikari.maximum-pool-size` | 50 |
| Lettuce Redis 池 | `spring.data.redis.lettuce.pool.max-active` | 64 |
| HttpClient 池 | `app.inference.maxTotal / maxPerRoute` | 200 / 100 |

> 三个 Spring 线程池都用 `CallerRunsPolicy`：超过 max + queue 时把任务退回主线程同步执行，
> 这一层兜底保证不丢任务；上游 Resilience4j 通常已经把异常情况限流掉。

## 3. 推理 RPC 三层保护栈

| 层 | 实现 | 触发条件 | 失败行为 |
| --- | --- | --- | --- |
| ① 线程隔板 (Bulkhead) | 有界 `inferenceExecutor` 队列 200 | 队列满 | `CallerRunsPolicy` 退回主线程 |
| ② 超时 (TimeLimiter) | `resilience4j.timelimiter.instances.inference.timeoutDuration=3s` | 超过 3 s | 抛 `TimeoutException`，计入熔断器 |
| ③ 熔断 (CircuitBreaker) | 50 次滑动窗口，failureRate ≥ 50% 或 slowCallRate ≥ 60% | 触发 | OPEN 10 s 后半开探针 5 次 |
| ④ Fallback | `InferenceClient#fallback` 返回 `Result.empty()` | 任何上层异常 | 上层走 `RecommendationService.fallbackCandidates`：Redis 旧候选 → 热门商品 → 空 |

熔断器状态可通过 actuator 实时观测：

```bash
curl http://localhost:8080/actuator/circuitbreakers
curl http://localhost:8080/actuator/circuitbreakerevents/inference
```

## 3.5 InferenceClient AOP 注意事项

`@CircuitBreaker` / `@TimeLimiter` 都靠 Spring AOP 代理生效。**`InferenceClient` 内部不能写 sync wrapper 调用 `recommendAsync(...)`**，否则 self-invocation 绕过代理，保护栈全部失效。

修复后的约定：

- `InferenceClient` 只暴露 `recommendAsync(...)`。
- `RecommendationService` 直接 `inferenceClient.recommendAsync(...).join()`。
- 如果需要 sync API，必须放在另一个 Bean。

## 4. Python 推理内部并发

`algorithm/inference/executors.py` 提供两个线程池：

| 池 | 作用 | 默认大小 | 调整环境变量 |
| --- | --- | --- | --- |
| `recall_io_pool` | 四路召回并发（Redis + Milvus 都是 I/O 等） | 32 | `RECSYS_RECALL_IO_WORKERS` |
| `ranking_cpu_pool` | MMoE forward（CPU 密集） | 物理核数 | `RECSYS_RANKING_CPU_WORKERS` |

执行流程：

```
recommend(request)
  │
  ├── read_user_bundle(user_id)   ← 一次 pipeline 拿
  │     HGETALL feature:user:{u}
  │     LRANGE  feature:user:{click|cart|purchase}_seq:{u}
  │     GET     feature:user:last_geo:{u}
  │     ZREVRANGE feature:user:tag_pref:{u}
  │     合计 1 次 RTT, 替代原来的 6 次串行
  │
  ├── 提交 4 路召回到 recall_io_pool, 透传 bundle.last_geo / bundle.tag_pref
  │     vector / lbs / tag / cache  并行
  │     wait(timeout = REQUEST_TIMEOUT_SEC * 0.7)
  │     任意一路超时 / 异常 -> 标记空, 不影响其他路
  │
  ├── 合并 + 去重 + 过滤 excludeItemIds
  │
  └── 提交排序到 ranking_cpu_pool
        future.result(timeout = 剩余预算)
        超时 / 异常 -> 退化按召回分排序
```

总超时 `RECSYS_REQUEST_TIMEOUT_SEC`（默认 2.5 s）小于 Java 端 `TimeLimiter` 的 3 s，
让 Python 端先回 fallback，避免 Java 端先超时丢弃响应。

`deploy/start_inference.ps1` 支持参数覆盖：

```powershell
./deploy/start_inference.ps1 -RecallIoWorkers 64 -RankingCpuWorkers 8 -TimeoutSec 2.0
```

## 5. 用户管理 / 登录限流

### 5.1 登录失败计数

`LoginAttemptService` 在 Redis 维护两组键：

| 键 | 计数对象 | 阈值 | TTL |
| --- | --- | --- | --- |
| `auth:login:fail:ip:{ip}` | 单 IP | `ipMaxPerWindow=30` | `windowSec=300` |
| `auth:login:fail:user:{name}` | 单账号 | `userMaxFailures=5` | `userLockSec=900` |

登录入口判断顺序：

1. `isIpBlocked(ip)` → 429 `too_many_login_attempts`
2. `isUserLocked(username)` → 423 `account_locked`
3. 进入正常 BCrypt 校验
4. 任一失败：`recordFailure(ip, username)`；成功：`recordSuccess(username)`

### 5.2 强制下线（全 token 失效）

实现：
1. `AuthService.login` 签发 token 时把 `tokenId` 写入 `auth:user:tokens:{userId}` Set，TTL = access token TTL + 60s。
2. 触发强制下线的 4 个动作（`UserService` 中调用 `authService.forceLogoutAll(userId)`）：
   - 管理员 `adminSetStatus(userId, 0)` 禁用
   - 用户 `changePassword`
   - 管理员 `adminResetPassword`
   - 管理员 `adminSetRoles`（roles claim 已经写在旧 token 里，必须强制重登）
3. `forceLogoutAll` 遍历该 Set，把每个 tokenId 加入 `auth:blacklist:{tokenId}`（TTL = access token TTL），然后删 Set。

### 5.3 默认密钥启动校验

`JwtTokenProvider` 启动时检查 `app.auth.jwtSecret`：

- 含子串 `change_me` 在 **local profile** 下大声 warning 但允许启动；
- 在任何其他 profile (dev/staging/prod) 下**直接抛 IllegalStateException 拒绝启动**。

### 5.4 默认拒绝访问

`SecurityConfig.anyRequest().denyAll()`：未列入白名单的路径默认 403。新增 controller 必须在 SecurityConfig 显式登记。

## 7. 容量推算

> 假设单机 4 核 16 G、JVM `-Xmx4g`。

| 路径 | 单次成本 | 同时容忍 |
| --- | --- | --- |
| 推荐刷新（推理走通） | ~500-800 ms | inferenceExecutor 64 in-flight + queue 200 ≈ 264 |
| 推荐分页（命中 Redis） | ~10-20 ms | Tomcat 400 worker × 50 rps/worker ≈ 20k rps |
| 商品批量查询 | ~20-30 ms | 同上 |
| 登录 | ~50-80 ms（BCrypt） | 受 Hikari 50 + BCrypt CPU 限制 ≈ 200 rps |
| 行为埋点 | <5 ms（Nginx 直入 Kafka） | 由 Nginx 决定，本服务零参与 |

## 8. 故障注入与验证手段

| 故障 | 验证方式 |
| --- | --- |
| Python 推理 5 s 卡顿 | inference 接口 sleep 5s，观察 actuator `circuitbreakers/inference` 状态由 CLOSED → OPEN |
| Milvus 不可用 | `MilvusClient.available()=False`，仅向量召回空，其他路正常 |
| Redis 卡顿 | Lettuce 池 `max-wait=1s` 后报错；推荐接口降级返回热门 |
| MySQL 慢 | Hikari `connection-timeout=3s`，超时计入 fallback |
| 推理 OPEN 状态 | 期间所有推荐刷新直接走 fallback；actuator/metrics 中 `resilience4j.circuitbreaker.calls{kind="not_permitted"}` 上涨 |

## 9. 监控与可观测性

### 9.1 后端 Prometheus / Actuator

| 路径 | 内容 |
| --- | --- |
| `/actuator/prometheus` | Java 全量指标 (Tomcat / Hikari / Lettuce / Resilience4j / executor / `@Observed` span timing) |
| `/actuator/circuitbreakers` | 推理熔断器状态 |
| `/actuator/circuitbreakerevents/inference` | 推理熔断事件流 |

最小关注指标：

- `resilience4j_circuitbreaker_state{name="inference"}`
- `resilience4j_circuitbreaker_calls_total{name="inference", kind=successful|failed|not_permitted|timeout}`
- `executor_queue_size{name="inferenceExecutor"}` / `executor_active_threads{name=...}`
- `hikaricp_connections_active`、`hikaricp_connections_pending`
- `lettuce_command_completion_seconds`
- `http_server_requests_seconds{uri="/api/v1/recommendations/refresh"}`

### 9.2 Python 推理 Prometheus

`http://localhost:9000/metrics` 暴露：

- 通用 `http_requests_total / http_request_duration_seconds` (来自 instrumentator)
- 业务 `recsys_recall_channel_size{channel}` histogram
- 业务 `recsys_recall_channel_latency_ms{channel}` histogram
- 业务 `recsys_rank_latency_ms` histogram
- 业务 `recsys_fallback_total{reason}` counter (`recall_empty / ranking_timeout / ranking_failed`)

### 9.3 分布式 trace

- Java 端：Micrometer Tracing + OTEL bridge；`RecommendationService.refresh` `@Observed` 自动 span；OTLP 默认 disabled，本地仅做 header 传播。
- Java → Python：`TraceHeaderInterceptor` 在 `inferenceRestTemplate` 上注入 W3C `traceparent: 00-{traceId}-{spanId}-01`。
- Python 端：`TraceContextMiddleware` 解析 `traceparent`，把 `trace_id / span_id / request_id` 注入 logging。
- 响应头：`x-trace-id`（中间件设置 + Java actuator 默认带 `traceId` MDC）。
- 接 Jaeger / Tempo：打开 `otel.exporter.otlp.enabled=true` 并启动 collector 即可。
