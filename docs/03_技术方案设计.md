# 技术方案设计 / 概要设计

## 1. 技术选型

| 模块 | 技术 | 选择理由 |
| --- | --- | --- |
| 前端 | Vue 3 | 组合式 API 适合拆分瀑布流、埋点、推荐状态 |
| UI | Element Plus | 组件成熟，适合快速搭建电商管理与展示界面 |
| 状态管理 | Pinia | 轻量，适合维护用户、购物车、推荐列表 |
| HTTP | Axios | 与 Spring Boot REST API 集成简单 |
| 后端 | Spring Boot 3.x | Java 主流 Web 框架，生态完整 |
| 项目管理 | Maven | 与 Spring Boot、MyBatis-Plus 集成成熟 |
| ORM | MyBatis-Plus | CRUD 开发效率高，SQL 可控 |
| 数据库 | MySQL 8.x | 存储业务数据、离线训练数据、推荐快照 |
| 缓存 | Redis | 存储推荐候选、实时特征、曝光集合 |
| 消息 | Kafka | 行为日志入口，承接实时与离线路径 |
| 实时处理 | Flink | 消费 Kafka 并维护用户实时状态 |
| 离线计算 | Spark | CPU 密集型特征处理和样本构建 |
| 模型 | PyTorch | 实现双塔和 MMoE 灵活 |
| 推理 | TorchServe / FastAPI MVP | TorchServe 用于模型封装，FastAPI 可作为早期联调服务 |
| 向量库 | Milvus | 存储物品向量，支持 ANN 检索 |
| 日志采集 | Nginx + Lua | 前端行为直接写 Kafka，降低后端业务接口压力 |

## 4. 模块划分

| 用户 Store | 保存 userId、登录态或游客 ID |
| 购物车 Store | 保存加购商品 |
| API 模块 | 封装 Axios 请求 |
| 埋点模块 | 曝光监听、行为发送、去重 |

### 4.2 后端模块

| 模块 | 职责 |
| --- | --- |
| Controller | 暴露 REST API |

| Kafka Consumer | 消费行为数据并批量入库 |

### 4.3 数据链路模块

## 7. 数据泄露控制

必须遵守

- 标签时间戳 `Label_Time`
- 所有用于模型输入的行为统计、序列、聚合特征，其最大行为时间必须 `< Label_Time`
- 训练样本中不能使用 Label_Time 之后发生的点击、购买、曝光。

- 离线特征代码和在线特征代码共用 `algorithm/common/features`。
- 排序负样本使用曝光未点击数据时，只能在同一曝光窗口内判断。
## 7.5 高并发设计概览
详细内容见 `10_高并发设计.md`，本节给出总览。
| 维度 | 实现 |
| --- | --- |
| 隔板 (Bulkhead) | `inferenceExecutor` / `snapshotExecutor` / `kafkaFlushExecutor` 三个独立有界线程池 |
| 熔断 (CircuitBreaker) | Resilience4j 滑动窗口 50，failure ≥ 50% ?slow 。60% 触发，OPEN 10s |
| 超时 (TimeLimiter) | 推理 RPC 单次 3s 强制熔断；Python 内部 2.5s wait |
| HTTP 层 | Apache HttpClient5 池化。00 。/ 100 / host |
| DB / Redis 层 | Hikari 50 / Lettuce 64 active |
| 登录限流 | Redis 计数：IP 5min 30 次失。/ 单账。5 次失败锁 15min |
| Tomcat | 400 worker，accept-count 200，max-connections 8192 |
| 监控 | Spring Actuator + Resilience4j 指标 (`/actuator/circuitbreakers`) |

## 8. 降级策略

| 故障 | 降级方案 |
| --- | --- |
| Python 推理服务不可用 | 返回 Redis 旧候选；若不存在则返回热门商品 |
| Milvus 不可用 | 使用热门商品或缓存召回 |
| LBS 召回数据缺失 | 跳过 LBS 通道，不影响双塔和标签召 |
| 标签倒排缺失 | 跳过标签召回，或使用类目热门商品补齐 |
| Redis 不可用 | 后端直接查询 MySQL 热门商品 |
| Kafka 不可用 | 前端埋点请求失败不阻塞用户操 |
| MySQL 不可用 | 商品接口返回错误，推荐接口不写快照 |

## 9. 推荐特征 Source of Truth

在线推荐请求中，Redis 推荐特征的读取统一由 Python 推理服务负责：
职责边界

- Spring Boot 后端只负责鉴权、生。requestId、调用推理服务、查询商品展示字段、写推荐候选队列和快照

- Spring Boot 不读取 `feature:user:*`、`rec:tag:index:*`、`rec:lbs:index:*` 作为推理输入

- Python 推理服务负责读取用户实时特征、用户标签偏好、用户最近 geohash、LBS 倒排、标签倒排、缓存召回和热门候选。- 同一次推理请求内，Python 推理服务应尽量在一个逻辑读取窗口内完。Redis 读取，并在日志中记录 `featureSnapshotTime`。
这样可以避免后端传入的用户实时特征和推理服务自行读取的召回索引来自不同时间点。
## 10. 基础用户体系设计

### 10.1 技术选型

| 能力 | 技 | 说明 |
| --- | --- | --- |
| 认证 | Spring Security + JWT | REST API 无状态认 |
| 密码存储 | BCrypt | 只保存不可逆哈 |
| 权限 | RBAC 简化模 | MVP 支持 `USER`、`ADMIN` |
| Token 黑名单 | Redis | 用户退出或禁用时可。token 失效 |

### 10.2 后端新增模块

| 模块 | 职责 |
| --- | --- |
| AuthController | 注册、登录、退 |
| UserController | 当前用户信息、资料修改、修改密 |
| AdminUserController | 管理员用户查询、禁用、启用、重置密码、角色调 |
| AuthService | 密码校验、JWT 签发、登录日 |
| UserService | 用户资料维护 |
| JwtAuthenticationFilter | 解析 token 并写与 Spring Security 上下 |
| PasswordService | BCrypt 加密和校 |

### 10.3 推荐链路改动

- 登录用户：后端从 JWT 中解。`userId`，推荐接口可以不再信任前端传入的 `userId`
- 游客用户：前端继续生成临时游。ID，用于未登录浏览和埋点。- 用户登录后：前端将推。Store 中的 `userId` 切换为登录用。`userId`，后续行为沉淀到稳定账号。
