# 数据库设计
## 1. 设计原则

- 业务表和推荐系统表分开命名

- 业务表前缀使用 `biz_`。

- 推荐表前缀使用 `rec_`。

- 原始天池数据表保留原表名，便于导入。

- Redis 只存放在线高频访问数据，不作为最终离线数据源。

- Kafka 作为实时日志缓冲，不保存长期训练数据。

## 3. Redis 设计

### 3.1 Key 命名

| Key | 类型 | TTL | 说明 |
| --- | --- | --- | --- |
| `rec:candidate:{userId}:{scene}` | List | 24h | 用户排序后的待展示商品 |
| `rec:exposed:{userId}:{requestId}` | Set | 24h | 当前推荐批次已曝光商品 |
| `rec:cache:{userId}` | ZSet | 3d | 精排 Top 50 但未曝光的缓存召回候选 |
| `rec:cache:count:{userId}:{itemId}` | String | 3d | 单商品缓存召回次数 |
| `feature:user:{userId}` | Hash | 7d | 用户实时特征 |
| `feature:user:click_seq:{userId}` | List | 7d | 用户点击序列 |
| `feature:user:cart_seq:{userId}` | List | 7d | 用户加购序列 |
| `feature:user:purchase_seq:{userId}` | List | 30d | 用户购买序列 |
| `feature:user:tag_pref:{userId}` | ZSet | 7d | 用户实时标签偏好，member 与 `tagType:tagValue` |
| `feature:user:last_geo:{userId}` | String | 7d | 用户最近 geohash |
| `rec:tag:index:{tagType}:{tagValue}` | ZSet | 24h | 标签倒排候选，score 为标签召回基础键 |
| `rec:lbs:index:{geohashPrefix}` | ZSet | 24h | LBS 倒排候选，score 为附近商品基础键 |
| `item:detail:{itemId}` | Hash | 24h | 商品展示字段缓存 |
| `rec:hot:items` | ZSet | 24h | 热门商品降级集 |

### 3.2 候选队列策

- 推荐刷新 **原子**覆盖 `rec:candidate:{userId}:home`：实现走 Lua 脚本
  `DEL + RPUSH + EXPIRE` 一次性提交，避免并发刷新时新旧候选混合或 TTL 丢失
  （见 `RedisCandidateRepository.REPLACE_CANDIDATES_SCRIPT`）。

- 前端分页时后端从 List 按游标读取。

- 商品曝光后加入 `rec:exposed:{userId}:{requestId}`
- 全部候选曝光后触发刷新

- 候。TTL 固定 24 小时。
### 3.3 缓存召回策略

缓存来源

- 每次精排 Top 500 中未曝光的商品。
退场规则：

- 曝光后从缓存删除

- 超出缓存大小后删除最早进入的商品

- 同一商品最多被召回 10 次。

- 保存超过 3 天自动过期。
### 3.4 LBS 与标签召回索引?
LBS 索引

- 离线任务读取 `biz_item.item_geohash`，按 geohash 前缀写入 `rec:lbs:index:{geohashPrefix}`。
- Flink 从行为事件中读取 `user_geohash`，写入 `feature:user:last_geo:{userId}`
- 在线推理时根据用户 geohash 前缀读取附近商品候选。
标签索引

- 离线任务读取 `rec_item_tag`，写入 `rec:tag:index:{tagType}:{tagValue}`。
- Flink 根据用户行为更新 `feature:user:tag_pref:{userId}`
- 在线推理时取用户 TopN 标签偏好，读取对应倒排候选并合并。
