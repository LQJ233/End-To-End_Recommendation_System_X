## 2. 字段映射

## 3. 行为类型对齐

### 3.1 数据集。项目内部

| Tianchi behavior_type | 含义 | 项目 behaviorType |
| --- | --- | --- |
| 1 | 浏览/点击 | 1 (点击) |
| 2 | 收藏 | 2 (收藏) |
| 3 | 加购物车 | 3 (加购) |
| 4 | 购买 | 4 (购买) |
| 0 | 曝光 (数据集未采集) | 0 (仅在线 Nginx Lua 埋点产生) |

### 3.2 离线训练时的影响

| 任务 | 正样本 | 负样本 | 备注 |
| --- | --- | --- | --- |
| 召回 | 1/2/3/4 全部 | 全局热门池随机负采样 | 无 Tianchihi 缺失曝光无关 |
| 排序 CTR | 1, 4 | **无法做「曝光未点。**, 改用未交互商品负采样 | 由 Spark 任务 |
| 排序 CVR | 4 | 1 (点击但未购买) | 按 user_id+item_id 没有 b=4 即视为未转化 |

> 设计文档见 `06_详细设计.md` 写的"排序负样。 曝光但未点击"在离线训练时改为
> "未交互负采样", 在线 Flink 链路如果 Nginx 埋点跑起来后, 仍按照原文方案处理。

## 4. 配套修改的代

- `scripts/import_raw_data.py`: 默认 `--max-rows 10_000_000`, 写入 `tianchi_mobile_recommend_train_user` + `rec_behavior_log` + `biz_item`, 末尾自动派生 `rec_item_tag`.
- `scripts/sample_tianchi.py`: 本地开发抽 5000 用户全行。 输出 `..._sample.csv`.
- `data-pipeline/spark/offline_samples_job.py`: 适配"无曝光」场景, CTR 负样本改为未交互商品负采。 CVR 负样本改为点击未购买.
- `scripts/sql/01_schema.sql`: `rec_item_tag` 增加 `UNIQUE KEY (item_id, tag_type, tag_value)`, 允许 import 任务重复执行.

## 5. 标准执行步骤

```powershell
# 1. 下载并放置 CSV
#    data/raw/tianchi_mobile_recommend_train_user.csv
#    data/raw/tianchi_mobile_recommend_train_item.csv

# 2. (可选）本地开发先抽样
python scripts/sample_tianchi.py --users 5000
# 然后跑 `_sample 文件重命名为原文件名以便 import 脚本识别
mv data/raw/tianchi_mobile_recommend_train_user.csv data/raw/_full_backup.csv
mv data/raw/tianchi_mobile_recommend_train_user_sample.csv data/raw/tianchi_mobile_recommend_train_user.csv

# 3. 初始化 schemaa (含本次新增的 unique key)
./deploy/init_db.ps1

# 4. 导入 (最多 1000 万行行为)
python scripts/import_raw_data.py --max-rows 10000000

# 5. 离线样本构建 + 训练 + 物品向量发布
python -m algorithm.training.run_daily
```

`run_daily.py` 会依次执行：

1. `data-pipeline/spark/offline_samples_job.py` (默认也是 1000 万行上限)
2. `data-pipeline/spark/popularity_job.py` 更新热门度 3. `data-pipeline/spark/tag_lbs_index_job.py` 重建 Redis 倒排
4. `algorithm.training.train_recall`
5. `algorithm.training.train_ranking`
6. `algorithm.training.publish_item_embedding` (写 Milvus)

## 6. 行数估算 (CPU 本地)

| 行数 | 导入耗时 | Spark 样本构建 | train_recall (1 epoch) | train_ranking (1 epoch) |
| --- | --- | --- | --- | --- |
| 100k | ~1 min | ~30 s | ~2 min | ~3 min |
| 1M | ~10 min | ~3 min | ~20 min | ~30 min |
| 10M | ~90 min | ~25 min | 3-5 h | 5-8 h |

如果时间。 推荐先用 5000 用户抽样 (大约 30 万行) 跑通全链路, 再用 1000 万行做最终训练。

## 7. 常见问题

**Q: 导入很慢, 能否用 LOAD DATA LOCAL INFILE?**

可以. 在 `import_raw_data.py` 与 `executemany` 替换为预生成 csv + `LOAD DATA LOCAL INFILE` 大概能再快 5-10  在 MySQL 8.x 需要在 my.cnf 里启用：

```ini
[mysqld]
local_infile=1
```

**Q: 没有阿里云账号怎么填？**

Kaggle 上有 user 表的镜像: https://www.kaggle.com/datasets/suolyer/tianchi-mobile-recommend-train-user
item 表可以从社区 GitHub 仓库找到, 见 `docs/08_README.md` 的相关链接。

**Q: 训练完发现召回向量都差不多。**

10M 条 + 1 。epoch 不一定收。 。`--epochs` 调到 3-5.

**Q: item_id vocab 这么大不会浪费内存吗?**

`item_id` 默认 2M × emb_dim 16 = 32M 参数 。128 MB (float32). 16G 内存机器完全可以承受. 太小 (10 条 会导致。6× 哈希冲突, 模型上限被严重压。 折中可以设到 500k.

**Q: enforceTimeSplit 是怎么实现的？**

`algorithm/training/datasets.py._enforce_time_split` 在加载 parquetet 之后, 在 `feature_time < timestamp` 过滤违反 Time Split 的行, 并打印计。 Spark 生成样本时已经把 feature_time 设为 timestamp - 1, 一般不会触。 这一层是兜底防穿透。
